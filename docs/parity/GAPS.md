## A. Architectural Decisions That Gate Everything Else (P0)

These aren't implementation gaps — they're forks in the design that determine whether half the table below is "must fix" or "out of scope." Resolving them takes a 30-minute call each.

| ID | Decision | Why it's P0 |
|----|----------|-------------|
| **D1** | **Is the API ever supposed to edit `wiki/`?** Today the model can `wiki_propose_edit` (creates a JSONL queue entry) but cannot Write or Edit. If "no, query-only," then 9 rows below collapse from gaps to design intent. If "yes," ~5–8 new tools are needed (file read for any path under wiki/, file write, str_replace edit, marker-bounded edit for AUTO blocks, append-only edit for log.md, transactional multi-file edit for bidirectional links) plus admin-approval plumbing. | Determines whether G06–G09, G14, G18–G20 are gaps or non-gaps. |
| **D2** | **Should `mode="api"` stream tokens?** Today only `mode="claude-code"` (subprocess) streams via `/query/stream`. The default api mode does a single non-streaming `messages.create()` (`backend/providers/deep_query.py:84`) — 5–30s silent wait. CC sets a high UX bar. Switching to `messages.stream()` is a real refactor: the tool-use loop has to handle partial responses, SSE plumbing has to extend to the api path, and `OrchestratorResult` becomes incremental rather than terminal. | Determines whether G02 is a P1 fix (do it) or a P3 acceptance (document the wait). |
| **D3** | **Are admin/curator workflows (INGEST, LINT) a non-goal for the API or just deferred?** Today neither exists. CLAUDE.md describes them as mandatory for the project's identity ("AI maintainer of this organization's feature knowledge wiki", §1 line 17). If "non-goal — INGEST/LINT happen in the terminal via CC," that's fine but needs to be stated; the project description currently implies otherwise. | Determines whether ~10 rows below are intentionally absent or are P1 gaps. |

## D. Patterns

Reading across the rows, three clusters dominate:

1. **Editing-the-wiki cluster** (gated by D1): G07, G14, G16, G18, G19, G27 — 6 gaps all collapse to one question.
2. **Live-data-and-fresh-state cluster**: G01 (live Jira), G03 (compaction), G04 (session posture), G09 (large output continuation) — all are "the model is working from stale or truncated data without knowing it." High-confidence wrong answers are the user-impact pattern. P1 across the board.
3. **Conversational-UX cluster**: G02 (streaming), G06 (feedback prompt), G08 (clarification turns), G15 (tool trace visibility), G17 (in-flow approval) — all about "the api-mode UX is a worse version of CC's UX even when it returns the correct answer." Pilot perception will hinge on this cluster more than on correctness.

If forced to pick one cluster to invest in for pilot, **cluster 2** (live data + freshness) is the one that produces wrong answers; cluster 3 produces correct-but-clunky answers; cluster 1 is a one-time scope decision.
