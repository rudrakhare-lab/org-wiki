VISITOR MANAGEMENT PRD

Problem Statement:
Ensuring compliance and safety with dynamic policies can no longer be managed with
traditional pen and paper visitor management. Increase in trend of hybrid workplaces, with remote-first employees, has further added to the planning time required to host in-person
clients and visitors. 

Personas involved:
	•	Employee
	•	Visitor
	•	Invited
	•	Walk-in
	•	Receptionist
	•	Security Guard
	•	Admin

Designs:  

Visitor Management – Admin
Visitor Management - Employee+Visitor
 Note: Designs posted here are for reference only to provide context. Kindly refer to figma file for the complete designs

The Employee Persona
Use Case 1.0: As an employee/host I want to invite a visitor on the web browser
	•	Employee invites a visitor through the web browser from the employee dashboard through quick actions or a separate tab of visitor management
	•	Employee can either invite from an existing database or add a new visitor
	
	•	Host can enter the following details:
	•	Event Title: “Meeting with <Employee Name>”. This can be edited by the user. Number of characters in the field: No limit.
	•	Mandatory field
	•	Type of visit: Type of visit to be a dropdown with fields:
	•	 Default: Business guests, Personal guests, Contract employees, Potential New Hire, Delivery Personnel
	•	Other option. When someone selects on Other, a box will appear next to the dropdown where in the person can enter the purpose (free text box) - Not built
	•	These fields are configurable 
	•	Date: Scheduled date of the event. 
	•	Past date cannot be selected
	•	Future date can be selected
	•	Time: Duration of the event 
	•	Handled for time zones
	•	Follows the time zone of the user profile
	•	Location: Location of the event. At present, we would only support the existing offices stored in the respective organization’s database.
	•	Default: By default the location would be the office of the employee creating the invite and would appear as in the design -Not built
	•	Incase a user wants to change location: When clicked on the office, it would open a dropdown menu where-in the employee can select any other office location
	•	Once the user selects a location, the dropdown box would remain as is
	•	Add Guests: Visitors to be invited to the event. (Explained in detail below)
	•	Guest Permission: This allows the employee to let the visitor avail certain amenities like Parking, Wifi-Access → V2
	•	Add Room: Configurable field. Only available to organizations using our meeting room solution. (Explained in detail below) → V2
	•	Add Desk: Configurable field. Available only to organizations who would like to book a desk for the visitor. (Explained in detail below) → V2
	•	Add Parking: Configurable field. Available only to organizations who would like to book a parking spot for the visitor. (Explained in detail below) → V2
	•	Personal note to guests: Free text box which can be used by the user to either describe the event or write a note to the invitees
	•	Note for admin: This is a separate text which only the admin would be able to see where-in the user can enter requests for resources required, etc. 
	•	Content in the text box will be sent to the admin as email (configurable) 
	•	This text box would appear in the admin dashboard with the list of meetings -- V2
	•	Email sent to admin: would consist of the Event Title, Type of visit, Guest list, location of the event, note to guests, and the note to admins 

	•	Once the employee completes the form and clicks on “Confirm”, the email notification would be sent to the guests, front desk and admin
	•	This is explained in the “Notifications” section of the PRD
	•	Post confirmation, the user would be asked if they would like to add additional amenities like meeting rooms, desk, etc -- V2
	•	Once an employee clicks on finish, they will be redirected to the dashboard of visitor management where they can view today’s and upcoming invites. (Explained in detail below)

Use Case 1.1: EMPLOYEE CAN VIEW THE INVITE (DASHBOARD)
	•	The employee is notified through status in the dashboard or through email when a visitor accepts the invitation
	•	Email notification will be configurable
	•	Employee can view following status with respect to the invited visitors -
	•	Invited → Visitor has been sent the invite
	•	Accepted → Visitor has accepted the invitation
	•	Rejected → Visitor has rejected the invitation
	•	Digipass generated → Once the visitor fills all the necessary fields, employee is notified that the digipass is generated
	•	Profile Completed → Once the visitor fills all fields of information mandated by the organization, the employee is notified that the visitor has completed their profile. This is valid when digipass flow is disabled
	•	Digipass Scanned → Once visitor scans the digipass, this status is visible
	•	Checked-in → Once the visitor is checked-in by the front desk, the status of the visitor changes to checked-in
	•	In case the digipass flow is disabled, then the status of the visitor remains as Profile Completed.
	•	Post check-in by the front desk, it would change to Checked-in
	•	Once the visitor is checked into the premises at the front desk, employee is notified by email or desktop notification
Use Case 1.2: EMPLOYEE EDITS / CANCELS THE INVITE
	•	Post creation of the invite, the employee can edit/cancel the event “n” minutes prior to the event. “N” → configurable time limit
	•	On cancellation of an event, all resources associated with that event would be canceled. For example, desks, meeting rooms, etc
	•	Edit Invite: Employee can edit multiple fields in the invite. Behavior of the system on change of each fields is as described below - 
	•	Event Title: Title of the event can be changed
	•	Date and Time: Event date and duration can be changed
	•	Personal note to guests / admin: Both these fields are editable
	•	Add / Remove Guests
	•	On change of the following fields, the guest is notified of the change through an email (mandatory) - 
	•	Date of the event: All guests
	•	Time of the event: All guests
	•	Personal note to guests: All guests
	•	Adding of a new guest: Invite email, only to the new guest added
	•	Removing of a guest: Canceled invite email, only to the guest removed
	•	On change of the following fields, the employee is asked if they would like to send emails to the guest - 
	•	Event Title
	•	On change of following fields, the guest is not notified - 
	•	Type of visit
	•	Note to admin
	•	Cancel Invite:
	•	Invite canceled sends email to all the guests informing them of the cancellation
	•	Employee can view this invite on their dashboard
	•	All the guests would be marked as canceled against the status in dashboard

Use Case 1.3: DOWNLOAD OF GUEST LIST

	•	When the number of guests in a invite crosses the threshold of 5 guests, then the employee can view the guests details only through download
	•	This downloaded file consists of the following headers:
	•	Name: Name of the visitor
	•	Email ID: Email ID of the visitor
	•	
Use Case 2.0: ADDING ONE GUEST AT A TIME

	•	New visitor: Employee has to enter basic information of the visitor like name, email address and phone number. 
	•	All fields can be made configurable as either optional or mandatory based on requirements of the organization. To note: If phone number is optional then email should be mandatory and vice versa. This is to ensure that we have some security check in-place
	•	Existing user: User should be able to search the visitor through the name or email address of the visitor. In case there are similar names, the search would show multiple results for the same name, with different email IDs 

Use Case 2.1: ADDING GUEST VIA BULK UPLOAD

Figma: https://www.figma.com/file/fCZKv6zvpTlvUWLyl7ZM8y/Visitor-Management?node-id=7217%3A292974&t=jG0PxlXFsGm2YnIk-4


	•	The employee can add visitors in bulk through an option – “Bulk Upload”
	•	This feature is controlled by a configuration - “Guest_Bulk_Upload”. 
	•	Its default value is set to “True” always
	•	It is to be added in the default configuration list of visitor management
	•	Employees can upload only 100 guests at a time.
	•	How it works: 
	•	When employee clicks on “Bulk Upload” → 
	•	A modal opens up which can be used to download the sample file and upload the details
	•	The “Upload” CTA should be grayed out until the user uploads a file successfully
	•	If the user clicks on “X” CTA, the user is redirected to the “Invite screen” without any action of bulk upload
	•	The employee can download the excel file for sample upload. Following fields are to be present in the file (All the fields mentioned below pertain to the visitor’s profile)
	•	Name: Name of the visitor
	•	Email ID: Email ID of the visitor
	•	Country code: The country to which the phone number belongs to
	•	Phone number: Phone number of the visitor
	•	Any other field that the admin would like to add to capture the profile of the visitor - V2 Kavya S( Broken flow - let’s pick it up in upcoming sprints)
	•	The employee then chooses the file to upload which has necessary information
	•	On upload, the system verifies if the file and all the information added in the file is in a consumable format
	•	If it is consumable → “Successfully Uploaded” is seen
	•	If any of the fields in the file is not consumable → A message - “There was an error while uploading the file” is seen. Further error / conflict states are explained below
	•	Once uploaded, the CTA changes to “Submit”. When the employee clicks on “Submit”, a pop-up opens with a message – “This would create the profile of the guest in the system. Do you want to proceed?”  
	•	If employee clicks on “Proceed” → profile of the guest is created (please refer to upcoming points for more information on profile creation)
	•	If employee clicks on “Decline” → the user is navigated to the modal where they can click on the “Submit” CTA
	•	If an employee drops off at this stage, i.e., if they click on the “x” CTA, then the profile of the guests would not be created. Previously uploaded file would not be stored
	•	Once the guests are created / uploaded, they are added to the invite and the following screen is seen   
	•	Mandatory fields
	•	In case a field is mandatory, then the words “Mandatory” should appear in the header of the field in the excel file
For e.g., if email ID is mandatory, then the header in the file would be “Email ID (Mandatory) If phone number is not mandatory, then the header in the file would be “Phone number”
	•	This would be applicable to all the fields that the admin would like to capture in the “visitor’s profile” that they would define for their organization
	•	Error / Conflict States
	•	If the file uploaded is not in a consumable format → “There was an error while uploading the file” is seen 

	•	If the file uploaded has more than 101 rows (100 guests + 1 header) → “Only 100 guests can be uploaded at a time. Please update the file and try again”
	•	If the file uploaded has conflicts → 
	•	This occurs when the information in the fields are not in consumable format or is wrong
	•	“There was an error while uploading the file” is seen with the number of conflicts called out 
	•	The employee can then download the file to check the erroneous rows and upload the file again
	•	Conflicts in the file can occur for the following reasons and their corresponding error message is to be present in the downloaded file in a separate column:  
	•	Email ID: “The Email ID format is wrong”
	•	Phone number entered does not match the number of digits as required by that country: “Phone number should have X digits as per the country code entered”
	•	Country code: If the country code does not exist → “This country code does not exist”
	•	Profile Creation
	•	On completion of the bulk upload action, the profile of the visitor is created in the system
	•	In case the visitor already exists in the database, then the following steps occur -
	•	A new profile of the visitor is NOT created. Existing profile continues.
	•	In case any of the information in non-mandatory fields is to be updated against the visitor’s profile, then the corresponding information will be updated based on the information in the uploaded file
	•	



APPROVAL WORKFLOW

Objective: According to office policies of some clients, it is required to review the visitors invited by an employee before allowing them inside the premises.

Prerequisite and Acceptance Criteria: 
	•	Employee should have a manager defined in their profile
	•	

Figma: Visitor Management – Figma
There are different types of emails for approve / reject / auto approve / auto reject. Please go through them all

Use Case 1.0: The employee’s manager needs to approve the invite sent to the visitor within a certain period of time

	•	Once the employee clicks on the “Invite Visitor” CTA, the invite opens where there is a banner which says “This invite requires approval from your manager”
	•	The employee fills the form and clicks on “Confirm” CTA, the following actions occur - 
	•	Employee is redirected to the dashboard where they see the event invite 
	•	The invite’s status would be in “Waiting for Approval” state
	•	Manager receives an email for approving the invitation
	•	None of the visitors receive the email invitation i.e., visitors booking is in “Pending” state
	•	If the manager approves the event invite within a certain defined time
	•	Employee is notified through an email that their invitation has been approved
	•	The status of the invite changes to “Approved” state in the dashboard
	•	Manager receives an email notification that they have approved the invite
	•	The visitors in the invite receive the email invitation i.e., visitors booking is in “Confirmed” state
	•	If the manager denies the invitation within a certain defined time
	•	The manager has to enter a rejection note along with the rejection
	•	The status of the invite changes to “Rejected” state in the dashboard along with the note
	•	Employee is notified through an email that their invitation has been rejected with a note from the manager
	•	Manager receives an email notification that they have rejected the particular invite
	•	The visitors do not receive any notification i.e., visitors booking is in “_____” state
	•	Manager:
	•	When the manager receives the email, they can click on either “Accept” or “Reject” button
	•	Once they click on the CTA in the email, the user will be redirected to the browser where the manager would be authenticated on Workinsync
	•	In case of “Accept” invite, once it is authenticated, the manager will see the success message “The invite has been successfully approved”
	•	In case of “Reject” invite, once it is authenticated, the manager would have an option to enter the rejection note. Once rejection note is entered, the message “Invite has been rejected” would appear
	•	Front desk: Until the invite is approved by the manager, the front desk would not see the invite. Once the invite is approved, then the front desk would see the visitors invited


Use Case 1.1: In case the manager does not approve the invitation within the defined time, there should be some action taken on the invitation

	•	In case the manager does not take any action on the invite within the defined time -
	•	The invite gets auto-approved (as a default configuration)
	•	An organization can also set it to “auto-reject”
	•	The above 2 options should be configurable properties with auto-approval being the default flow when approval workflow is setup

BOOKING A PARKING FOR VISITOR
Problem Statement:
	•	The employees / admin are not able to inform the visitors if there’s parking that they can avail when visiting their office
	•	In case parking is available in the organization, currently, employees are not able to check the availability or book the same for visitors
Figma: https://www.figma.com/file/EfaWNaNGp8PQZGIObVOpEa/VM---Employee%2Bvisitor?type=design&node-id=1310-235597&mode=design&t=6GiHaFrOfDYub40C-0   Acceptance Criteria:
	•	Create ‘Visitor’ tag
	•	VMS-Parking Configuration
	•	Property Name: Visitor_Parking_Auto_Allocation
	•	True → Enable Parking Selection in Visitor Invite
	•	False → Disable Parking Selection in Visitor Invite
	•	Default: False
	•	Use Parking Module to allow for auto-allocation of parking bookings for visitors
	•	For allocation parking slots to visitors, consider the Visitor tagged slots in the whole premise for that office
	•	Do not allow employees to view Visitor tagged parkings
	•	Only allow visitor tagged parking slots to be considered and allocated for visitors
	•	Parking should be booked even if registration number is required (only for visitors)
	•	There would be no changes to premises page, parking report (do not consider visitor parkings in this page)
Use Case 1.0 As an employee I want to be create an invite and book a slot for a visitor in the invite form so that I can assign a parking slot to a visitor and provide them a dedicated parking slot 
	•	They open the invite form and fill out the details of the event
	•	When property (Visitor_Parking_Auto_Allocation) is true, there will be an “ + Parking” button above the list of added guests.
	•	The button should be disabled if there are no visitors added to the invite, button only gets enabled when at least 1 visitor has been added
	•	‘Added Guests’ title to be added to distinguish the added visitors
	•	The format of visitor details will also be altered to have Name, Email and phone number in subsequent lines, the rest of the format stays the same
	•	The order in which visitors are added to the invite form will be reserved, going from bottom - top to top - bottom.
	•	When “ + Parking” is selected, a modal will open
	•	Modal:
	•	Title: Add Guest Parking
	•	First line: Added guests & parking slot chips with the number of slots per vehicle type (only car and bike for now)
	•	List of visitors added with CAR and BIKE selection chips next to their details - none selected by default
	•	Cancel button: if selected should close the modal and go back to the invite form
	•	Done button: Saves the selections for each visitor
	•	If there is no chip selected for a visitor, treat it as “0” slot selected
	•	Car chip selected = 1 car slot selected, Bike chip selected = 1 bike slot selected
	•	In the create state - when the host selects Car/Bike for a visitor, the count should decrease for the vehicle type. When the host deselects Car/Bike, then the count should increase fir that vehicle type.
	•	Only show Car or Bike IF there are Visitor tagged parkings for that visitor type. Ex. If there are 0 visitor tagged bike parkings, do not show the Bike option, only show Car. - Not Built
	•	If there are 0 slots available for that vehicle type, the Car/Bike chip should be disabled
	•	Once the host clicks on Done, the parking selections would get saved - not created in backend - and a parking symbol - P - would show up against the visitor on the invite form. This symbol would only appear for those visitors who had parking selected for them.
	•	The employee would proceed to Confirm the Invite
	•	Once the invite has been confirmed, there should be a check in the system whether the parking slots are still available at the time of selecting ‘Confirm Invite’
	•	Hence there would be a delay in triggering the invite email immediately after the confirm invite button is selected
	•	If even one slot is not available for one visitor at the time of checking the system for slot allocation, there would be a toast message displaying “Parking Slots Unavailable. Please Try Again.”
	•	The screen would be taken back to the editing state of the invite
	•	After confirming the invite, the employee can see the assigned zone, level and parking slot in the invite preview 
	•	Invite Email will be sent to the visitor
Use Case 1.1 As an employee I want to be able to edit my invite and visitor parkings - Not Built
	•	The employee is allowed to edit the invite
	•	Only when they click on OFFICE or TIME
	•	A modal should tell the visitor that “Making this change will reset visitor parking(s). Would you like to proceed?”
	•	Back to editing and proceed buttons
	•	Back to editing  will take them back to editing the form in the previous prefilled state
	•	Proceed would cancel the previous parking slots and reset the dropdown selections back to “Select Vehicle”
	•	The new selection of Office, date and time should also be displayed
	•	The employee will then have to go through the same flow as use case 1.1
	•	In the current scope we will not let employees edit the parking of individual visitors
	•	In the case where an employee selects OFFICE, DATE, TIME but doesn't choose a different value and selects the same value → no modal should show up

Solution:
	•	Reducing the overall number of emails that get triggered to the invited visitor
	•	Simplifying the invited visitor registration flow
	•	Cater to the new ‘no digipass’ flow
Acceptance Criteria:
	•	The digipass should be controlled property specific to a particular office
	•	Property for digipass -VISITOR_DIGIPASS

	•	‘Accept Invite’ button to expire just before the event starts - allowing the visitor to change their details multiple times before the visit (already exists)
	•	New Tab: Complete Profile Flow
	•	First Screen: where the visitor will verify their personal details
	•	There will be a collapsible card for the Event Details
	•	Addition of the country code drop-down for phone number (replaces the old design of writing the country code country)
	•	Visitor will have to check if their personal details are filled in correctly
	•	Behavior of Email ID, Phone Number fields:
	•	When Email ID and Phone Number have been input by the host at the time of invite creation → visitor cannot edit email ID, phone number, they will be ‘Read Only’
	•	Field should be in disabled text state on the screen
	•	When host does not enter the phone number during the time of invite creation → visitor can write the phone number
	•	Field should be in enabled text state on the screen
	•	Behavior of other fields
	•	Name, Phone Number is mandatory
	•	Company, Designation would be non-mandatory fields
	•	Email would already be input - mandatory
	•	Only when the visitor fills in mandatory fields will the ‘Next’ button be highlighted
	•	Second Screen: where the visitor will upload their profile and photo ID

	•	Digipass Email - Second Email
	•	Subject line: Welcome to [Company Name]: Your Digital Access Pass is Here!

	•	Flow remains the same for invited visitor

Use Case 4.0 Making profile ID configurable for invited visitors
Problem Statement:
	•	Certain organizations do not make it mandatory for invited visitors to upload their profile ID during the profile completion process
Solution:
	•	Make the uploading of a profile ID in the invited flow registration process a configurable option
Use Case 4.1 Invited visitor is completing their profile (Profile ID upload is disabled)
	•	Once the invited visitor clicks on Confirm and Accept Invite email → they would enter their basic details and confirm the invite → the user is redirected to complete their profile

Dynamic Fields
Dynamic fields are fields in a database that have different data types or values for different records. They are used to store data that vary from one record to another and that does not fit neatly into a predetermined data structure. They can be helpful to store data in a flexible format that can adapt to changing needs

Different types of dynamic fields include:

(Text in yellow were part of Feb 2023 sprint’s scope and have been developed by backend)

	•	Text fields: Text fields can be used to store text strings of varying lengths. They might be used to store strings of varying lengths like short descriptions (comments), longer descriptions (notes, etc)
	•	Boolean fields: These fields can be used to store True/False values. They might be used to store things like flags or indicators
	•	Numeric fields: These fields can be used to store numbers including integers, decimals, and floating point values. They might be used to store things like prices, quantities or ratings
	•	Data fields: These fields can be used to store dates and times. They might be used to store things like event dates, timestamps, etc
	•	Object fields: These are for storing complex data like arrays or nested objects. They might be used to store things like lists, trees or graphs
	•	Geospatial fields: These fields can be used to store geographical coordinates and other geospatial data. They might be used to store things like addresses, locations or maps
	•	Binary fields: Fields to store binary data such as images, audio or video. They might be used to store things like attachments, media files, digital documents etc

Parameters to define dynamic fields:

A user can define a dynamic field based on the following parameters:
	•	Title: The header of the dynamic field
	•	For instance, dynamic field for entering the information of the person who is creating the catering order would have label as “Organizer’s Contact Information”
	•	Data structure (or Subtype): The structure of the data inside the dynamic field can be defined by the user. 
	•	For instance, if one is storing customer information, they might want to create separate fields for Customer’s name, address and phone number
	•	Or if one is storing a date range, they may want to create separate fields for start date and end date
	•	Default value: The value that would be present by default in the field can be defined by the user creating the dynamic field
	•	Placeholder text: The user creating the data field may want to define a placeholder text to provide the user a format for entering the data
	•	Other things to consider:
	•	Data type: To consider the appropriate data type for each field, based on the type of data being stored. E.g., a numeric field for storing numbers, a text field for storing strings
	•	Data validation: Rules to ensure that only valid data is entered into the fields. E.g., only numbers are entered into numeric fields, validate that a date field contains a valid date, URL consists of a valid type of URL link, etc
	•	Min and max value: Certain numeric fields may have constraints like, minimum and maximum value that can be entered
	•	Maintainability: The option to add and remove fields as the data changes should be simple and scalable 
Working of these Dynamic fields:
	•	The dynamic fields will be setup by the admin (Out of scope)
	•	Applicable places: 
	•	Visitor persona: While uploading profile details such as image and documents, the visitor would see the fields to be filled
	•	Walk-in visitor (Receptionist persona): Once the receptionist has entered the basic details, then the fields are to be visible

Front Desk Dashboard - Version 2
Objective - To improve scalability and performance of the front desk dashboard by migrating all frontend logic (status, filtering, and sorting) to the backend.   Additionally, introducing pagination on front desk dashboard to help users with easy tracking and managing visitors and improve user experience


Front desk dashboard - Overview
The Frontdesk Dashboard is a comprehensive tool designed to help users manage and track visitors efficiently. It offers a range of filters, including office, date and time range, and status. These options enable users to search, sort, and filter visitors with ease, streamlining their workflow.
By clicking on a visitor's name, a side panel opens, providing detailed information about the selected visitor, ensuring all necessary data is readily accessible.


Problem Statement- The current front desk dashboard processes all visitor records, filtering, and status logic on the frontend. This results in:  
1. Performance Issues: As data volume increases, the frontend struggles to handle filtering and sorting, causing delays.
2. Scalability Challenges: Managing filtering, sorting, and pagination on the frontend limits the system's ability to handle larger datasets or add new features.  

Solution- 
	•	Search bar
	•	Functionality: Users can search for visitors using their name, email, or phone number.
	•	Default state: Empty
	•	Search Behavior:
	•	When users start typing any character, results are populated if the character appears anywhere in the name, not necessarily at the beginning. Example: If a user searches for "Harsha" by typing 'H', "Harsha" will appear first, followed by other names containing 'H' in the order they appear.
	•	Reset Behavior:
	•	If the page is hard reloaded or refreshed using the refresh icon, the search bar resets to its empty state.
	•	If any filter is changed on the front desk, the search bar also resets to its empty state.

	•	Cancel Toggle
	•	Purpose: Allows users to filter out canceled invites, showing only active ones. This helps front desk personnel focus on actionable invites.
	•	Default State: Disabled
	•	Disabled: Displays all invite statuses except 'Canceled'.
	•	Enabled: Displays all invite statuses, including 'Canceled'.
	•	When the “Office,” “Date,” or “Time Range” filters are applied, the toggle resets to its default state (disabled). If the toggle is manually enabled, other filters remain unaffected.
	•	When the “Status” filter is applied, the toggle becomes grayed out (disabled). Users can enable or disable it only after the status filter is reset.
	•	If the user applies the "Canceled Invites" filter, the toggle automatically enables.
	•	When the page is refreshed (auto or via the refresh icon), the toggle resets to its default state (disabled). 
	•	Office Filter
	•	Purpose: Lists all offices for a particular build.
	•	Default Value: Default office
	•	If a specific office is selected, it remains unaffected by other filters or status changes.
	•	The selected value persists across page refreshes but resets to the default office on a hard reload.

	•	Date Picker
	•	Purpose: Filters invites by the selected date.
	•	Default Selection: Current date.
	•	If the date is changed and the office filter is altered, the date resets to the current date.
	•	Selected values persist across refreshes but reset to the default state on a hard reload.

	•	Time filter
	•	Purpose: Filters invites by a specific time range.
	•	Default Value: 12:00 AM - 11:59 PM.
	•	If altered and the date filter is changed, the time filter resets to its default state.
	•	If altered and the office filter is changed, the time filter resets to its default state.
	•	Selected values persist across refreshes but reset to the default state on a hard reload.

	•	Refresh Icon
	•	Purpose: Reloads the page and resets specific components.
	•	Opens the first invite with its detailed side panel view.
	•	If the side panel is closed, refreshing the page reloads it entirely and opens the first invite.
	•	Interactions:
	•	Cancel Toggle: Resets to its default state (disabled).
	•	Other filters are not affected.


	•	Type of Invite 
	•	Purpose: Displays the type of workflow (invite, walk-in, self-check-in) and the type of visitor (e.g., business guests, vendors, delivery personnel). 
	•	Entry and Exit time 
	•	Purpose: Displays the start and end times of invites.
	•	Default Order: Upcoming invites appear first, followed by others in chronological order.

	•	Status
	•	Provides information about the visitor's status.
	•	Status includes - UPCOMING, CHECKED-IN, SELF CHECK-IN, QR CHECK-IN, OVERSTAY, CHECKED-OUT, QR CHECK-OUT, SELF CHECK-OUT, PENDING, CANCELLED, DECLINED, RECEPTION CHECKED-OUT, AWAY FROM PREMISES, SECURITY CHECK-IN, SECURITY CHECKED-OUT, DELAYED CHECK-IN, CONFIRMED, REJECTED, SIGNED_IN, SIGNED_OUT, EXPIRED
	•	Default State: None selected.
	•	Filter Options: Only displays statuses available for the current invites. For example, if invites only have statuses "Upcoming," "Expired," and "Pending," only these options appear in the dropdown.
	•	Interaction with office filter - If the status filter is applied and office filter is altered, the status filter goes back to its default status
	•	Interactions
	•	Office Filter: Altering the Office filter resets the Status filter to its default state.
	•	Date Picker: Altering the Date filter resets the Status filter.
	•	Time Filter: Altering the Time filter resets the Status filter.
	•	Search Bar: Results are displayed only if they align with the applied filter and visitor searched. Otherwise, no results are shown.
	•	Refresh Icon: Resets the Status filter to its default state
	•	Cancel Toggle: If the Status filter is applied and the Cancel toggle is enabled, only invites matching the selected status and canceled invites are displayed.
